class="k">\subsection</span><span class="nb">{</span>Basic Entropy-based Algorithm<span class="nb">}</span></div><div class='line' id='LC62'><br/></div><div class='line' id='LC63'>The first approach to optimizing the set of questions is to determine which ones give the highest quality information.  The selection problem we encounter can be framed in terms of a <span class="k">\textit</span><span class="nb">{</span>Total Utility Function<span class="nb">}</span> (TUF) that quantizes the total uncertainty in the database.  Selecting tokens for labeling by the crowd corresponds to a reduction of the TUF and we seek to select those tokens that do this maximally.  While strictly speaking the TUF can be any function of the probabilistic content of each token, we model our function on the total entropy of the system.  For an individual document, this is the entropy over the space of possible labelings <span class="k">\textbf</span><span class="nb">{</span>s<span class="nb">}</span>:</div><div class='line' id='LC64'><span class="k">\begin</span><span class="nb">{</span>equation<span class="nb">}</span></div><div class='line' id='LC65'>H(d<span class="nb">^{</span>i<span class="nb">}</span>) = <span class="k">\sum</span><span class="nb">_{</span><span class="k">\textbf</span><span class="nb">{</span>s<span class="nb">}}</span>p<span class="nb">^{</span>i<span class="nb">}</span>(<span class="k">\textbf</span><span class="nb">{</span>s<span class="nb">}</span>)log(p<span class="nb">^{</span>i<span class="nb">}</span>(<span class="k">\textbf</span><span class="nb">{</span>s<span class="nb">}</span>))</div><div class='line' id='LC66'><span class="k">\end</span><span class="nb">{</span>equation<span class="nb">}</span></div><div class='line' id='LC67'>Since all documents in the database are independent, the total entropy is equivalent to the sum of the entropies of each document <span class="s">$</span><span class="nv">\sum</span><span class="nb">_{i}H</span><span class="o">(</span><span class="nb">d^{i}</span><span class="o">)</span><span class="s">$</span>.  For a document of length <span class="s">$</span><span class="nb">N</span><span class="s">$</span> in a label space of size <span class="s">$</span><span class="nb">L</span><span class="s">$</span> there are <span class="s">$</span><span class="nb">N^{L}</span><span class="s">$</span> possible values that <span class="s">$</span><span class="nv">\textbf</span><span class="nb">{s}</span><span class="s">$</span> may take, making direct calculation intractable for any nontrivial task.  We can bypass this issue by remembering that absolute values of the system entropy are not needed and we care only about relative changes in response to selecting one token over another.  </div><div class='line' id='LC68'><br/></div><div class='line' id='LC69'>We consider our selection problem to be one of partitioning the set of all tokens <span class="s">$</span><span class="nb">T</span><span class="s">$</span> into those selected for submission and those not.  Let <span class="s">$</span><span class="nb">X</span><span class="s">$</span> be the set of tokens pushed to the crowd and <span class="s">$</span><span class="nb">Y</span><span class="s">$</span> be the remaining set of tokens such that <span class="s">$</span><span class="nb">Y </span><span class="o">=</span><span class="nb"> T </span><span class="o">-</span><span class="nb"> X</span><span class="s">$</span>. For a fixed budget, we seek to select <span class="s">$</span><span class="nb">X</span><span class="s">$</span> such that <span class="s">$</span><span class="nb">H</span><span class="o">(</span><span class="nb">Y|X</span><span class="o">)</span><span class="s">$</span> is minimized.  The equation for conditional entropy states from Section~<span class="k">\ref</span><span class="nb">{</span>sec:uncertainty<span class="nb">}</span> states</div><div class='line' id='LC70'><span class="k">\begin</span><span class="nb">{</span>equation<span class="nb">}</span></div><div class='line' id='LC71'>H(Y|X) = H(X,Y) - H(X)</div><div class='line' id='LC72'><span class="k">\end</span><span class="nb">{</span>equation<span class="nb">}</span></div><div class='line' id='LC73'>The first part of the right hand side, <span class="s">$</span><span class="nb">H</span><span class="o">(</span><span class="nb">X,Y</span><span class="o">)</span><span class="s">$</span>, is a constant since <span class="s">$</span><span class="nb">X </span><span class="nv">\cup</span><span class="nb"> Y </span><span class="o">=</span><span class="nb"> T</span><span class="s">$</span>.  Thus minimizing <span class="s">$</span><span class="nb">H</span><span class="o">(</span><span class="nb">Y|X</span><span class="o">)</span><span class="s">$</span> is equivalent to maximizing <span class="s">$</span><span class="nb">H</span><span class="o">(</span><span class="nb">X</span><span class="o">)</span><span class="s">$</span>, the entropy of the selected tokens.  If selection is done on-line one token at a time, <span class="s">$</span><span class="nb">H</span><span class="o">(</span><span class="nb">X</span><span class="o">)</span><span class="s">$</span> is the marginal entropy of an individual token and top-k selection is equivalent to selecting the top-k marginal entropies from <span class="s">$</span><span class="nb">T</span><span class="s">$</span>.  Explicitly, the marginal entropy for token <span class="s">$</span><span class="nb">i</span><span class="s">$</span> is:</div><div class='line' id='LC74'><br/></div><div class='line' id='LC75'><span class="k">\begin</span><span class="nb">{</span>equation<span class="nb">}</span></div><div class='line' id='LC76'>H<span class="nb">^{</span>i<span class="nb">}</span> = -<span class="k">\sum</span><span class="nb">^{</span>L<span class="nb">}_{</span>j=1<span class="nb">}</span>p<span class="nb">^{</span>i<span class="nb">}_{</span>j<span class="nb">}</span>log(p<span class="nb">^{</span>i<span class="nb">}_{</span>j<span class="nb">}</span>)</div><div class='line' id='LC77'><span class="k">\end</span><span class="nb">{</span>equation<span class="nb">}</span></div><div class='line' id='LC78'><br/></div><div class='line' id='LC79'><span class="k">\begin</span><span class="nb">{</span>figure<span class="nb">}</span></div><div class='line' id='LC80'>		<span class="k">\includegraphics</span><span class="na">[width=0.48\textwidth]</span><span class="nb">{</span>images/ent<span class="nb">_</span>dist1.png<span class="nb">}</span></div><div class='line' id='LC81'>		<span class="k">\label</span><span class="nb">{</span>fig:ent<span class="nb">_</span>dist<span class="nb">}</span></div><div class='line' id='LC82'>		<span class="k">\caption</span><span class="nb">{</span>Typical entropy distribution for a document in <span class="k">\sysName</span><span class="nb">}</span> </div><div class='line' id='LC83'><span class="k">\end</span><span class="nb">{</span>figure<span class="nb">}</span></div><div class='line' id='LC84'><br/></div><div class='line' id='LC85'>where the <span class="s">$</span><span class="nb">p_{j}</span><span class="s">$</span> is the marginal probability of label <span class="s">$</span><span class="nb">j</span><span class="s">$</span> being the correct label.  Marginal probabilities are calculated using the CRF-variant of the Forward-Backward algorithm <span class="k">\cite</span><span class="nb">{}</span>.  The marginal entropy provides a formulation for a token&#39;s individual certainty or uncertainty in its label.  Compare the distributions in Figure~<span class="k">\ref</span><span class="nb">{}</span>.  When the distribution is fairly peaked and we are certain of an answer, the entropy tends to be fairly low.  A more uniform distribution, however, leads to a higher entropy.</div><div class='line' id='LC86'><br/></div><div class='line' id='LC87'><span class="k">\subsection</span><span class="nb">{</span>Reducing the Space of Questions<span class="nb">}</span></div><div class='line' id='LC88'><span class="k">\subsubsection</span><span class="nb">{</span>Constraining Tokens Per Document<span class="nb">}</span></div><div class='line' id='LC89'>While we may sort all of the tokens in the database by entropy and select the top-k, this may lead to less than optimal results.  Individual tokens are not independent.  Modifying the probabilistic label attribute of one token in a document may invoke additional corrections when the inference algorithm is re-run.  We discuss this constrained inference idea further in the context of data integration in Section~<span class="k">\ref</span><span class="nb">{}</span>.  Suffice to say, because we can&#39;t anticipate the expected gain from each individual token, we choose not to risk the redundancy of batching multiple high entropy tokens from the same document.</div><div class='line' id='LC90'><br/></div><div class='line' id='LC91'>The motivation for this choice can be found in Fig~<span class="k">\ref</span><span class="nb">{</span>fig:ent<span class="nb">_</span>dist<span class="nb">}</span>, which shows the entropy distribution for a typical document.  The highest entropy values appear in isolated neighborhoods which correspond to the segmentation boundaries.  Since constrained inference primarily supplies additional correction the neighborhoods of constrained tokens, the probability of two selected tokens sharing the same &quot;correction window&quot; is high.  </div><div class='line' id='LC92'><br/></div><div class='line' id='LC93'>Thus the strategy implemented is to select only a single token from each document, that token being the one with the highest entropy.  The explicit process can be found in Figure~<span class="k">\ref</span><span class="nb">{}</span> and uses a simple hash map with a docID key to assert only one token per docID is allowed.  All tokens are cycled through in a linear fashion.  If the current docID is not represented in the map, we add it, while if it is represented we only add it if it&#39;s entropy is higher than the token currently in the map.  With each document being associated with its highest entropy token, for the remainder of this discussion we refer to selection of a token and selection of document interchangeably.</div><div class='line' id='LC94'><br/></div><div class='line' id='LC95'>The existence of such a &quot;correction window&quot; as mentioned above brings up an interesting tradeoff that should be considered.  When the two differ, is it better to select a single highly uncertain token or a token at the center of a high entropy neighborhood, none of whose entropy is as high as the first?  As an additional entropy metric for selection, we introduce the concept of &quot;neighborhood entropy&quot;, that is the entropy associated with a token and its nearest neighbors.  It represents a balance between the per-token uncertainty and the expected benefit of crowdsourced answer to its neighbors.  We compare both entropy metrics in the experiments of this paper. </div><div class='line' id='LC96'><br/></div><div class='line' id='LC97'><span class="k">\begin</span><span class="nb">{</span>figure<span class="nb">}</span></div><div class='line' id='LC98'>		<span class="k">\includegraphics</span><span class="na">[width=0.48\textwidth]</span><span class="nb">{</span>images/red.png<span class="nb">}</span></div><div class='line' id='LC99'>		<span class="k">\label</span><span class="nb">{</span>fig:cluster<span class="nb">}</span></div><div class='line' id='LC100'>		<span class="k">\caption</span><span class="nb">{</span>PLACEHOLDER.<span class="nb">}</span> </div><div class='line' id='LC101'><span class="k">\end</span><span class="nb">{</span>figure<span class="nb">}</span></div><div class='line' id='LC102'><br/></div><div class='line' id='LC103'><span class="k">\subsubsection</span><span class="nb">{</span>Clustering Similar Tokens<span class="nb">}</span></div><div class='line' id='LC104'>Individual documents, especially citation data, may contain significant overlap between authors, conferences, publishers, etc. that appear in more than one document.  While we accept a certain redundancy for quality control purposes, this should be tightly controlled through the AMT interface.  Selecting the same tokens with the same labels from different documents essentially doubles the cost for a single answer.</div><div class='line' id='LC105'><br/></div><div class='line' id='LC106'>Our solution to this problem is through a novel clustering technique.  Tokens that share similar text and machine labeling properties have a high probability of sharing the same true labels.  By mapping multiple tokens into the same <span class="k">\textit</span><span class="nb">{</span>Question Cluster<span class="nb">}</span>, a single answer can be used to modify labels for a large number of tokens.  Whereas before we constrained the token space to the set of documents, here we constrain the space even further to only the set of unique clusters.  In the rest of this section we describe three specific cluster sets of cluster properties.</div><div class='line' id='LC107'><br/></div><div class='line' id='LC108'>All cluster sets require tokens belonging to the cluster to share the same token text and machine labeling, but are set apart by different neighborhood constraints.  Context is important because it&#39;s possible that two tokens with the same text such as the word &quot;computer&quot; could actually require different labelings and it would be false to cluster them together.  The different cluster properties trade off accuracy and cluster size and we compare them in the experiments section.  Figure~<span class="k">\ref</span><span class="nb">{</span>fig:cluster<span class="nb">}</span> compares two similar tokens and illustrates the different cluster properties that are checked.</div><div class='line' id='LC109'><br/></div><div class='line' id='LC110'><span class="k">\textbf</span><span class="nb">{</span>Same Field:<span class="nb">}</span> The strictest of the cluster properties.  It promotes the least amount of clustering, but contains greatest accuracy.  The initial cluster representative defines a field by its CRF max likelihood label and all preceeding and suceeding tokens with the same label.  Tokens are only added to the cluster if they share the entire field.</div><div class='line' id='LC111'><br/></div><div class='line' id='LC112'><span class="k">\textbf</span><span class="nb">{</span>Same Label Neighbhorhood:<span class="nb">}</span> A relaxing of the Same Field properties to only compare labels one position before and after the token.  The existence of sub-entities such as a city that appears in more than one Proceedings field or an author that appears with different groups of authors motivate this approach of not checking the entire field.  Clustering is greatly increased, but at the expense of a small amount of misclassification.</div><div class='line' id='LC113'><br/></div><div class='line' id='LC114'><span class="k">\textbf</span><span class="nb">{</span>Same Token Neighborhood:<span class="nb">}</span> The simplest of the cluster properties.  We don&#39;t check any of the machine labelings, but measure redundancy based purely on the text associated with each token.  Tokens are clustered together that share the same token as well as the same token directly preceeding and suceeding it.  This has the advantage of being purely dependent on the data and not at all on the CRF output.</div><div class='line' id='LC115'><br/></div><div class='line' id='LC116'><span class="k">\sean</span><span class="nb">{</span>One paragraph about implementation of pseudo-code as database procedures somewhere.<span class="nb">}</span></div><div class='line' id='LC117'><br/></div><div class='line' id='LC118'><span class="k">\subsection</span><span class="nb">{</span>Ordering Questions<span class="nb">}</span></div><div class='line' id='LC119'><br/></div><div class='line' id='LC120'>The result of running one of the above clustering algorithms reduces the selection problem into one of selecting the <span class="k">\textit</span><span class="nb">{</span>cluster<span class="nb">}</span> which provides the maximum reduction of uncertainty in the database.  We consider three different ways of ranking clusters for top-k selection.  Adhering to our information theoretic framework, it&#39;s possible to select the &quot;representative&quot; high entropy token used to initiate each cluster and rank by that token&#39;s individual entropy.  On the other hand, if clusters are largely skewed in size, it may be more beneficial to rank by the actual size of the cluster.  As a final heuristic attempting to combine both the entropy and cluster size approaches, we can sort by the total entropy of each cluster, that is, the sum of entropies of every token in the cluster.  Our experiments compare and contrast these different ordering techniques. </div></pre></div>
